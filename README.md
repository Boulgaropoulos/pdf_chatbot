# PDF Document QA

A small full-stack application for uploading PDF documents, processing them asynchronously, and asking grounded questions with page-level citations.

## What It Does

- Create named workspaces/projects.
- Upload one or more PDFs into a workspace.
- Process documents in a background worker.
- Track document/job status as `queued`, `processing`, `ready`, or `failed`.
- Ask questions about uploaded documents.
- Answer only from retrieved document chunks.
- Return citations with filename and page number.
- Say when the uploaded documents do not contain an answer.

## Tech Choices

- **Next.js + React + TypeScript** for a simple frontend flow.
- **FastAPI** for a small, explicit REST API.
- **PostgreSQL + pgvector** for metadata and semantic vector search in one database.
- **PyMuPDF** for PDF text extraction while preserving page numbers.
- **OpenAI `text-embedding-3-small`** for embeddings.
- **OpenAI `gpt-4o-mini`** for concise grounded answers.
- **Docker Compose** to run frontend, backend, worker, and database together.

I chose a modular monolith because the assignment values clear data flow and scoping. The API and worker share the same backend codebase and Docker image, but run as separate processes.

## Architecture

```text
Browser
  |
  | HTTP
  v
Next.js frontend
  |
  | REST API
  v
FastAPI backend  -----> PostgreSQL + pgvector
  ^
  |
Worker process
queued jobs -> parse PDF -> chunk text -> embed chunks -> store vectors -> ready/failed
```

## Run Locally

Requirements:

- Docker Desktop
- An OpenAI API key

Create an environment file:

```powershell
Copy-Item .env.example .env
```

Edit `.env` and set:

```env
OPENAI_API_KEY=your_openai_api_key_here
```

Start everything:

```powershell
docker compose up --build
```

Open:

```text
Frontend: http://localhost:3000
API docs: http://localhost:8000/docs
Health:   http://localhost:8000/health
```

## API Overview

```text
GET /workspaces
```

Lists existing workspaces.

```text
POST /workspaces
```

Creates a named workspace. The frontend stores the selected workspace id in `localStorage`.

Example request:

```json
{
  "name": "Contract review"
}
```

```text
POST /workspaces/{workspace_id}/documents
```

Uploads one or more PDFs, stores them, creates queued processing jobs, and returns immediately.

```text
GET /workspaces/{workspace_id}/documents
```

Lists uploaded documents and their current processing status. The frontend polls this endpoint.

```text
GET /jobs/{job_id}
```

Returns one processing job and its status/error message.

```text
POST /workspaces/{workspace_id}/chat
```

Retrieves relevant chunks from ready documents and asks the LLM to answer from those chunks only.

Example response:

```json
{
  "answer": "The refund period is 30 days.",
  "citations": [
    {
      "filename": "policy.pdf",
      "page": 7
    }
  ]
}
```

## Data Flow

1. User creates or selects a workspace.
2. User uploads PDFs.
3. Backend saves each file and creates a `Document` plus `ProcessingJob`.
4. Worker claims queued jobs.
5. Worker extracts page text with PyMuPDF.
6. Text is split into overlapping chunks while preserving page numbers.
7. Chunks are embedded with OpenAI embeddings.
8. Chunk text, page number, document id, and vector are stored in PostgreSQL.
9. User asks a question.
10. Backend embeds the question and retrieves similar chunks using `pgvector`.
11. LLM receives only the retrieved chunks.
12. Backend returns the answer and builds citations from retrieved chunks.

## Honesty And Citations

Citations are not generated by the LLM. They are created by the backend from the chunks retrieved from PostgreSQL, using the stored filename and page number.

If retrieval finds no strong match, or if the LLM says the context does not contain the answer, the API returns:

```text
I couldn't find an answer to that question in the uploaded documents.
```

## Tradeoffs

- No authentication. A browser-local workspace id is enough for the assignment scope.
- No OCR. Scanned PDFs without embedded text will fail with an error message.
- No WebSockets. The frontend uses polling for status updates.
- Citations are page-level, not exact text spans.
- Chat history is kept in frontend state and is not persisted.
- Database tables are created on startup instead of using a migration tool.
- The worker uses simple polling instead of Redis or a dedicated queue.
